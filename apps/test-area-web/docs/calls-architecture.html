Calls + Bookings · architecture

Phase 0 doc. Read this before any code lands. Push back on anything that doesn't make sense — every architectural call is reversible at this stage and locked in once we start migrations.

How to use this doc: read it once top-to-bottom. Disagree with something? Say so before phase 2 (schema). Once tables exist with data in them, changing them costs migrations and downtime.

1. What this slice ships

End-to-end booking + calling between a caller (consumer) and a callee (professional). Concretely:

  1. Caller browses a professional's available slots and books one (POST /bookings).
  2. Booking creation reserves wallet money for the call price + platform fee.
  3. At call time both parties hit POST /calls/:id/join and get an Agora RTC token.
  4. They join the channel, talk, hang up.
  5. Backend settles the wallet (split between pro + platform), records strikes if a side no-showed, fires outbox events.
  6. Cancellations, no-shows, mid-call disconnects, network flaps all have explicit handling.

Out of scope: reviews/ratings (fast-follow slice), recording, group calls, scheduled-then-rescheduled flow. Reschedule is cancel-and-rebook for now.

3. Two fee modes

The platform fee is configurable via wallet.fee_mode in platform_config. Both modes always pay the platform when the pro is paid — full refund is the only path where platform earns nothing.

Mode A — deduct_from_payee (default)

The price the user sees is what they pay. Platform fee comes out of the pro's earnings.

Pro's hourly rateCaller paysPro receivesPlatform earns
₦400₦400₦360 (90%)₦40 (10%)

Journal at booking (call_payment_reserve): pending_debits_pool +400, user_wallet -400.
Journal at settlement (call_settlement): payee_wallet +360, platform_revenue +40, pending_debits_pool -400.

Mode B — add_to_payer

The pro's rate is what the pro receives. Platform fee is added on top at checkout.

Pro's hourly rateCaller paysPro receivesPlatform earns
₦400₦440₦400 (100%)₦40 (10%)

Journal at booking (call_payment_reserve): pending_debits_pool +440, user_wallet -440.
Journal at settlement (call_settlement): payee_wallet +400, platform_revenue +40, pending_debits_pool -440.

How the booking remembers which mode it used

Critical: the booking row snapshots fee_mode_used at creation time. A config flip mid-booking-lifecycle does NOT retroactively re-split. The settlement reads the snapshot, not the live config. Same principle as bank_snapshot on withdrawals.

The booking row stores three amount fields, derived from the rate + mode at booking time:

  • total_paid_kobo — what the caller's wallet was actually debited.
  • payee_amount_kobo — what the pro receives on full settlement.
  • platform_fee_kobo — what platform_revenue gets on full settlement.

total_paid_kobo = payee_amount_kobo + platform_fee_kobo always holds.

4. Bookings vs Calls — separate tables

Decision: separate. A booking is "an intent to call at time T"; a call is "an actual session that happened (or didn't)". Reasons:

  • A booking has a 1:1 with a call only on the happy path. Cancellation removes a booking; no call ever existed. Reschedule (post-launch) creates a new booking from an old one.
  • Different lifecycles: bookings have cancel_window and a single transition. Calls have join events, leave events, status flips, partial settlement.
  • Querying "show my upcoming bookings" vs "show my call history" want different shapes.

A call row is created when the booking confirms (i.e. when wallet reservation succeeds), not at start_at. The call sits in scheduled status until both sides have joined.

6. Money flows by terminal state

Reservation amount = total_paid_kobo. Whatever doesn't go to the pro goes back to the caller as a refund. Whatever doesn't go to the caller goes to the pro and to platform_revenue (split per fee_mode + connected fraction).

Platform always gets paid when the pro gets paid. Platform's share scales with whatever fraction the pro is paid: if the pro receives 100% of their cut, platform gets 100% of the fee; if pro is pro-rated to 50%, platform gets 50% of the fee. Full refund = 0 fee.

Terminal state Caller refund Pro receives Platform receives Strike
completed (both stayed) 0 full payee_amount_kobo full platform_fee_kobo —
completed (pro-rated, connected ≥ min_billable) (1-frac) × total_paid frac × payee_amount frac × platform_fee —
disconnected_caller (1-frac) × total_paid frac × payee_amount frac × platform_fee caller (if config on)
disconnected_callee (1-frac) × total_paid frac × payee_amount frac × platform_fee pro (if config on)
no_show_caller (pro waited) x% × total_paid y% × payee_amount y% × platform_fee caller (if config on)
no_show_callee (caller waited) full total_paid 0 0 pro (if config on)
no_show_both full total_paid 0 0 —
cancelled_outside_window full total_paid 0 0 —
cancelled_inside_window (caller cancels) (1-penalty%) × total_paid 0 penalty% × total_paid (cancellation fee → platform_revenue) —
cancelled_inside_window (pro cancels) full total_paid 0 0 pro (if config on)

Where frac = connected_seconds / scheduled_seconds, and the no-show x% + y% = 100% by config constraint.

Note on cancellation_inside_window penalty destination

When the caller cancels inside the window, the kept penalty goes to platform_revenue (not the pro), because no service was rendered. The pro is incentivized via the "no-show by caller" path: if they want compensation they need to actually show up and wait. This is a deliberate choice — open to argument, but I think it's right.

7. Strikes

A strike is a black mark against a professional that counts toward an automatic ban threshold. Configurable triggers; configurable threshold; configurable dispute window. The pro can dispute, admin reviews.

Triggers (each independently toggleable)

  • professional.strike_on_no_show — pro didn't show at start time + grace.
  • professional.strike_on_late_cancel — pro cancelled inside the cancellation window.
  • professional.strike_on_mid_call_quit — pro disconnected mid-call and didn't re-join within network-flap window.

Strike lifecycle

   active ─── (pro disputes within strike_dispute_window_days) ───► disputed
     │                                                                   │
     │                                                                   │ (admin reviews)
     │                                                            ┌──────┴──────┐
     │                                                            ▼             ▼
     │                                                          upheld        voided
     │                                                       (counts toward  (does not count)
     │                                                        ban threshold)
     │
     │ (admin manually voids without dispute)
     └───────────────────────────────────► voided

Only active and upheld strikes count toward the ban threshold. disputed doesn't count until resolved (so a pro under review isn't suddenly auto-banned by the next strike landing).

Auto-ban

When a strike lands and total counting strikes ≥ professional.strikes_before_ban, the user gets flagged account_status = 'suspended' automatically. Admin can lift via the existing user moderation endpoints (already in profile / admin features). Pro sees a message on profile screen explaining the ban + what to dispute.

Endpoints (Calls slice ships these)

  • GET /me/strikes — pro lists their own strikes (active, disputed, upheld, voided), shows count toward ban + remaining-before-ban.
  • POST /strikes/:id/dispute — pro disputes within window, body: { comment: string }. Sets status disputed.
  • GET /admin/strikes — admin lists all strikes, filterable by status / professional / reason.
  • POST /admin/strikes/:id/uphold — admin upholds a disputed strike (final).
  • POST /admin/strikes/:id/void — admin voids a strike. Body: { reason: string }. Final.

11. Agora — what we use, what we choose

  • RTC. Yes. Audio + video.
  • Tokens. Server-issued, short-lived (~1 hour), scoped to (channel, uid, role=publisher). Backend mints via the App Certificate. Mobile never sees the certificate.
  • RTM (signaling). No. We don't need it — call lifecycle is driven by our backend's join/leave endpoints + Agora's webhooks for ground truth.
  • Cloud recording. No, deferred. Adds cost + legal complexity. Revisit when reviews/disputes demand it.
  • Webhooks (Agora notifications). Yes — for ground-truth "did they actually join the channel?" Backed up by client-reported join/leave so we have two sources.
  • Channel naming. call_<call_id>. UUID-shaped, globally unique. Caller UID = stable hash of caller_user_id, callee UID = stable hash of callee_user_id (Agora UIDs are 32-bit ints).
  • Token expiration. 1 hour from issuance. Mobile renews via POST /calls/:id/renew-token 5 min before expiry.

12. Lifecycle — who decides "the call is over"?

Three sources of truth, in priority order:

  1. Both sides explicitly leave via POST /calls/:id/leave. Cleanest.
  2. Agora webhook says channel is empty — both sides disconnected (clean leave or network drop). Triggered after Agora's own ~30s grace.
  3. Scheduled end + grace — if duration ran out and we haven't heard from anyone, we end it ourselves.

The settlement fires once, idempotently, the moment the call is determined to have ended. Idempotency key on the settlement journal is call:<call_id>:settle. Whichever signal arrives first wins; the others are no-ops.

13. Time tracking — connected_seconds

connected_seconds = max(0, min(caller_left_at, callee_left_at) - max(caller_joined_at, callee_joined_at))

i.e. the window where both were in the channel simultaneously. If one party joins, sits alone for 30s, the other joins for 5 min, then both leave: connected = 5 min. The lonely 30s isn't billable.

Network flaps are tricky. If a side disconnects and rejoins within network_flap_window_seconds, we treat it as continuous (no gap deducted). If beyond, it's a disconnect — we settle as disconnected_X with whatever connected time we recorded.

14. Idempotency keys

  • Booking creation: POST /bookings accepts Idempotency-Key header. Same key → same booking.
  • Reservation journal: call:<call_id>:reserve (existing wallet flow convention).
  • Settlement journal: call:<call_id>:settle.
  • Refund journal: call:<call_id>:refund:<reason> where reason is e.g. cancel, no_show_callee, etc.
  • Strike creation: strike:<call_id>:<reason_code> — same call + same reason can't double-strike.

15. Outbox events

New event types to add to OutboxEventType:

  • booking.created
  • booking.cancelled
  • call.scheduled — fired at booking time so mobile can subscribe.
  • call.starting_soon — fired by a cron N min before start_at.
  • call.completed — fired on final settlement.
  • call.no_show — fired on no-show terminal state.
  • strike.issued — pro gets an email.
  • strike.disputed — admin gets an email.
  • strike.resolved — pro gets the outcome.

The outbox worker email branch handles each. Push notification dispatch comes with §9 (notifications slice).

16. Cron / background jobs (new)

  • Call starter cron. Every 30s, find calls with start_at <= now() && status='scheduled', flip to waiting_for_parties, fire call.starting_soon outbox event.
  • No-show resolver cron. Every 30s, find calls with status='waiting_for_parties' && start_at + grace < now(), evaluate join state, transition to terminal.
  • Stuck-call resolver cron. Every minute, find calls with status='in_progress' where neither side has reported activity in N minutes; force-end via flow 12.3.

17. Reverse-engineering checklist (read this when something breaks)

  1. Look up the booking. fee_mode_used, total_paid_kobo, payee_amount_kobo, platform_fee_kobo.
  2. Look up the call. Status, joined/left timestamps, connected_seconds, settlement_journal_id, refund_journal_id.
  3. Read call_events top-down. The lifecycle is here.
  4. Pull the journals: reservation, settlement, refund. Sum to caller_paid - pro_received - platform_received - caller_refunded = 0.
  5. Outbox: did the side-effect emails fire?
  6. Agora dashboard: channel logs match our records?

18. What I'm choosing and why — push back here

The decisions you should disagree with before phase 2 if they're wrong:

  • Bookings + calls as separate tables. Could be one table with nullable join columns. Separate is cleaner; "one big row per intent" gets ugly when reschedule lands later.
  • Snapshot fee_mode + amounts on bookings. Mid-flight config flip risk is real. Worth the storage.
  • Connected_seconds as the billable basis. Alternative: scheduled duration. Connected is more honest but punishes bad luck (network).
  • Cancellation penalty → platform_revenue, not pro. Argument for pro: they cleared their slot. I'm choosing platform because no service was rendered. Reversible.
  • Auto-ban on threshold without admin review. Strict. Alternative: queue for admin review. I think auto-ban + easy admin lift is right.
  • Server-side Agora token issuance. Industry standard. Could be App-ID-only (no token), but that's prod-unsafe.
  • Duration locked at booking, not extendable. If users want a longer call they hang up and rebook. Simpler.
  • No reschedule in this slice. Cancel + rebook covers it. Reschedule = future feature.
